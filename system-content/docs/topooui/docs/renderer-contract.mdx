---
title: "Renderer Contract"
description: "How schema types map to system-owned templates and documentation blocks."
---
Schema alone is not enough.

Every page type and block type needs a system-owned renderer.

## Mapping Principle

The content repo says:
- what the content is
- what fields it contains

The system says:
- how that content becomes UI

## Current Renderer Direction

- `board-landing` -> `BoardLandingTemplate`
- `component-index` -> `ComponentIndexTemplate`
- `docs-system-index` -> `DocsSystemIndexTemplate`
- `doc-page` -> `DocPageTemplate`
- `docs-block-doc` -> `DocsBlockDocTemplate`

## Why This Matters

Without a renderer contract, content repos can still end up controlling layout by embedding MDX structure directly.

With a renderer contract:

- content stays structural
- visuals stay centralized
- system upgrades do not require editing content repos

## Current Constraint

This contract is partly documented and partly transitional.

The next implementation step is to make pages such as:

- `/docs/topooui`
- `/docs/topooui/components`
- `/docs/topooui/docs`

fully template-driven instead of layout-driven MDX pages.
